### 为什么效果最好？

我问了一个关键问题：**"为什么AMP用的人少，但能做出最好的效果？"**

答案是：**Skill 库**

什么是 Skill？就是 Claude 的"工作指南"。当我加载 `frontend-design` Skill 时，Claude 不再是个通用AI，而是变成了一个"遵循设计规范的高级工程师"。

```bash
# 加载 Skill
amp skill add anthropics/skills/frontend-design
```

Claude 瞬间懂了：
- 毛玻璃怎么写（backdrop-filter）
- 3D 变换怎么优化（will-change）
- 手势库怎么选（Hammer.js vs Pointer Events）
- 性能怎么保证（requestAnimationFrame 节流）

**直译**：提示词 + Skill = 一次到位

### 调试过程的真相

但老实说，**并不是第一次就完美**。

[截图位置2 - 第1个Thread的问题：3D球体出现抖动]

**第1个Thread**：球体抖动
- 问题：手势拖动时，Three.js 的 Euler 角旋转有万向锁（Gimbal Lock）
- Claude 的建议：用四元数（Quaternion）代替
- 耗时：1个Thread，15 分钟

[截图位置3 - 第2个Thread的问题：节点标签遮挡]

**第2个Thread**：标签遮挡问题
- 问题：人物标签和关系线重叠，看不清
- 方案：动态标签（鼠标悬停时显示，平时隐藏）
- 耗时：1个Thread，20 分钟

[截图位置4 - 第3个Thread的问题和最终优化]

**第3个Thread**：性能优化
- 问题：100+ 个节点时，帧率下降
- 方案：使用 LOD（细节等级，Level of Detail）和 GPU 实例化
- 耗时：1个Thread，30 分钟

**总耗时：3个Thread，约2小时**

### 关键洞察：为什么需要多个Thread？

这里很重要。AMP 虽然最终效果最好，**但它不是"一次到位"的工具**。它更像是**"和顶级工程师配对编程"**。

每个 Thread 都是一次深度对话：
- 我发现问题 → 描述症状
- Claude 分析根因 → 给出方案
- 我反馈修改效果 → 微调参数

这个过程很像**真实的工程工作流**。所以虽然花时间多，但你学到的东西最多。

### 总结：选AMP的人

✅ 想要**最高品质**的界面  
✅ **愿意在细节上投入时间**  
✅ **想通过合作学到工程思维**  
✅ 有**一定开发经验**的人

❌ 预算极其紧张  
❌ 只想快速出原型（不关心品质）

**成本**：有费用（Claude API 按使用量计费）

## 关键洞察：有一个对所有 AI 应用者都重要的方法论

到这里，大多数文章会说"选择 AMP，完美"。

但我想分享一个更深层的发现。

### 一个问题：为什么 AMP 的效果最好，但花的时间最多？

因为 **AMP 鼓励迭代和优化**。

我在开发过程中，用了 3 个不同的 Thread：
1. 第1个 Thread：提交最初的设计需求 → 获得第一个可用版本
2. 第2个 Thread：指出问题（标签遮挡、性能不足）→ 深度优化
3. 第3个 Thread：微调细节 → 最终精品

这个过程让我意识到一件事：**每个 Thread 都在不断完善"提示词"本身**。

### PDCA 循环：一个在 AI 时代必须学会的能力

让我把这个过程总结成一个框架：

```
第 1 轮：Plan（规划）
  ↓
  💭 初始提示词：「用 Three.js 做一个红楼梦人物关系图谱」
  
第 1 轮：Do（执行）
  ↓
  🤖 Claude 按照提示词生成代码
  
第 1 轮：Check（检查）
  ↓
  👁️ 我测试效果：「有抖动、有遮挡、性能不足」
  
第 1 轮：Act（行动）
  ↓
  🔧 修改提示词：加入「四元数旋转」「动态标签」「LOD优化」等关键词
  
────────── 分割线 ──────────

第 2 轮：Plan（规划）
  ↓
  💭 改进后的提示词：「用四元数... 用动态标签... 用 LOD...」
  
第 2 轮：Do（执行）
  ↓
  🤖 Claude 按照改进的提示词重新生成
  
第 2 轮：Check（检查）
  ↓
  👁️ 我测试效果：「好多了，但还可以优化颜色」
  
... 循环继续 ...
```

这就是传说中的 **PDCA 循环**（Plan-Do-Check-Act）。

### 这为什么重要？

因为在 AI 时代，**你的核心价值不在于"懂代码"，而在于：**

1. **能清楚地描述问题** - "标签遮挡"
2. **能识别真正的根因** - 不是"标签位置不对"，而是"标签和线重叠了"
3. **能提出解决方案** - "用动态标签"，而不是"改个颜色"
4. **能不断迭代需求** - 每轮都让提示词变得更精准

### 大佬们怎么说这件事？

**Andrej Karpathy**（前Tesla AI主管，现OpenAI研究员）说过：

> "The machine learning engineers of the future won't be writing code. They'll be writing prompts."
>
> 未来的机器学习工程师不会写代码，而是写提示词。

但他后来补充了一句更重要的话：

> "And the best prompt engineers are not those who write perfect prompts on the first try. They're the ones who iterate, fail fast, and learn from every attempt."
>
> 最好的提示词工程师不是那些一次写出完美提示词的人，而是那些敢于迭代、快速失败、从每次尝试中学习的人。

**这和什么有关？PDCA。**

---

### 我在这次对比测试中学到的核心能力

**不是**："哦，AMP 最好，我以后就用 AMP。"

**而是**："在 PDCA 循环中，我不断优化了对需求的理解，以及对'如何和AI合作'的理解。"

这个能力，迁移到任何工具、任何项目上都适用。

### 怎么把这套方法用到自己的工作中？

**第1次开发时：**
```
初始提示词 v1.0（简单粗糙）
  ↓
产出 v1.0（有问题）
  ↓
修改提示词 v2.0（加入解决方案）
  ↓
产出 v2.0（好多了，但还有坑）
  ↓
修改提示词 v3.0（精细化需求）
  ↓
产出 v3.0（接近完美）
```

**开发完成后（这是关键）：**
```
整理 最终提示词 v3.0
  ↓
写成文档，保存
  ↓
下一个项目时，直接用 v3.0 作为基础模板
  ↓
省时 60%
```

### 小白最容易忽视的一点

很多人做完一个项目就算完了。

**但最聪明的做法是：在项目完成后，把最终版本的提示词沉淀成"工作模板"。**

为什么？因为：
- 你的下一个项目会 80% 相似
- 你不用重新发明轮子
- 你对这个领域的理解会越来越深
- **这就是个人竞争力的积累**
